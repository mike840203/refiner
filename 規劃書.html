部署到公司 NAS

三種部署方案

目前雛形是純前端、資料存在「使用者自己的瀏覽器（localStorage）」。要多人共用，重點是把資料從各自瀏覽器搬到 NAS 集中儲存。

方案 A · 最快

NAS 當靜態網頁

把 index.html 丟進 NAS 的網頁服務（Synology Web Station / QNAP Web Server），同事用區網網址開啟。

  • ✅ 半天可上線、零開發
  • ⚠️ 資料仍各存各的瀏覽器，無法共用同一份
  • 適合：個人使用、示範、過渡
方案 B · 推薦

NAS 集中儲存

NAS 上以 Docker 跑一個輕量後端（Node/Express），資料存 SQLite 或 MariaDB（NAS 內建）。全公司看同一份、即時同步。

  • ✅ 真正多人協作、單一真相來源
  • ✅ 可加帳號、權限、通知
  • 需求：NAS 支援 Container（多數 Synology+/QNAP 皆可）
方案 C · 過渡

共享資料夾 + 匯入匯出

把匯出的 JSON 放 NAS 共享資料夾（Synology Drive），需要時匯入。

  • ✅ 立即可用、零開發
  • ⚠️ 非即時、需人工同步，易衝突
  • 適合：2–3 人、低頻率更新
建議路線：先用 方案 A 讓團隊試用、確認流程，同步開發 方案 B 作為正式版。方案 B 完成後才有「帳號與主管專屬功能」的基礎。
Synology 實作流程

怎麼在 NAS 上把容器跑起來

公司 NAS 經實測為 Synology DSM 7.x，內建 Container Manager（即 Docker）。以下是從零到上線的實際步驟，分「快速版」與「共用正式版」。

✅ 已實測：HHYNAS（192.168.10.250，主檔案伺服器）與 TimNas（192.168.10.230）皆為 Synology DSM 7.x。建議先用 TimNas 當測試機，確認無誤再上正式機。

前置（一次性）

  1. 安裝 Container Manager：DSM → 套件中心 → 搜尋「Container Manager」→ 安裝。（若搜不到，代表此機型不支援 Docker，需改用方案 A 或換機。）
  2. 建立資料夾：File Station → 建立 docker/launch-tracker/web，把 index.html 放進 web 內。
  3. 取得管理員帳號：安裝套件、建立專案都需要 NAS 管理員權限。

路線一 · 快速版（靜態網站，約 1 小時）

讓全公司用同一個網址打開系統；資料仍各自存在各人瀏覽器。適合先試用、確認流程。

  1. Container Manager → 專案（Project） → 新增。
  2. 專案名稱填 launch-tracker；路徑選剛建的 /docker/launch-tracker。
  3. 來源選「建立 docker-compose.yml」，貼上下列內容：
services:
  web:
    image: nginx:alpine
    container_name: launch-tracker
    restart: unless-stopped
    ports:
      - "8080:80"
    volumes:
      - /volume1/docker/launch-tracker/web:/usr/share/nginx/html:ro
  1. 下一步 → 完成。Container Manager 會自動下載 nginx 映像並啟動容器。
  2. 全公司開 http://192.168.10.250:8080 即可使用；之後更新只要替換 web 內的 index.html。

路線二 · 共用正式版（方案 B，需先開發後端）

資料集中存在 NAS、全公司同一份、可加帳號與主管權限。需先把目前的純前端加上一個輕量後端（即 Roadmap 階段 2）。

架構：前端（瀏覽器）→ 後端 API（容器）→ 資料庫（SQLite 檔放 NAS volume，或 MariaDB 容器）。compose 形如：

services:
  app:
    image: launch-tracker:1.0      # 自建映像，階段 2 產出
    container_name: launch-tracker
    restart: unless-stopped
    ports:
      - "8080:3000"
    volumes:
      - /volume1/docker/launch-tracker/data:/app/data   # 資料留在 NAS

資料夾 data 位於 NAS 共享空間 → 自動納入 NAS 快照與備份；換新版本只需更新映像標籤、重新建置專案。

好記網址 + HTTPS（選用）

  1. DSM 控制台 → 登入入口 → 進階 → 反向代理伺服器：把 https://nas/launch 或一個子網域導向容器的 localhost:8080。
  2. 憑證：DSM 可套用 Let's Encrypt（需對外網域）或自簽憑證供內網使用，讓網址變 https。

維運與備份

  • 開機自動啟動：compose 設 restart: unless-stopped，NAS 重開機會自動拉起。
  • 備份：資料 volume 在共享資料夾 → 啟用 NAS 快照（Snapshot）與 Hyper Backup。
  • 監控：Container Manager 內可看容器 CPU/記憶體、日誌、重啟。

風險與對策

需要注意的事

資料安全與備份

階段 0/A 資料在本機，換機/清快取會遺失 → 養成定期匯出；方案 B 後由 NAS 統一備份（快照）。

多人同時編輯衝突

方案 C 以人工同步易衝突 → 盡早進入方案 B 由後端統一寫入。

NAS 單點故障

依賴 NAS 開機與網路 → 啟用 NAS 快照與離線匯出備援。

維護人力

自建工具需 1 名維護者 → 程式極簡、單檔好交接，並保留本規劃書與註解。

結論

需要主管的決定

雛形已可立即使用，零成本。為了讓全公司共用並加入主管專屬權限，建議核准進入方案 B。

① 核准試用

同意以方案 A 上 NAS，讓相關部門先試用 1–2 週。

② 核准階段 2

同意開發集中儲存版（方案 B），作為正式系統基礎。

③ 指派資源

確認 NAS 權限、容器環境，與每週可投入的開發時數。

一句話：用一個免費、單檔的看板，讓上市流程從「靠問」變成「打開就知道」。先試用，再決定要不要升級成全公司共用的正式系統。
